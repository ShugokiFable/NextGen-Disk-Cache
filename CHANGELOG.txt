NextGen Disk Cache 2.0.0 - narrow IAT hook, one FOMOD, unified versioning
============================================
Responds to an independent audit of 1.4.1 (2026-07-22). Accepted findings:

- HOOK SCOPE (accepted). 1.4.x attached Detours trampolines to CreateFileA/W,
  which intercepts every module in the process including unrelated SKSE
  plugins. Default is now an Import Address Table hook applied ONLY to
  SkyrimSE.exe's own imports, so other modules keep their own IAT and are
  never intercepted. New src/IatHook.cpp parses the PE import directory by
  hand (no DbgHelp import added). Process-wide Detours is retained behind
  iHookScope=1 for the Experimental profile.
  Verified by A/B test: with an auxiliary DLL opening the same archive with
  identical flags, Safe logged ONLY the executable's own open; Experimental
  logged both. If the executable has no importable slot the plugin logs
  "IAT hook was not installed; plugin left inert" and does NOT silently
  escalate to a process-wide trampoline.

- bRaiseMemoryPriority (accepted, my error). It sets MEMORY_PRIORITY_NORMAL
  (5), which Microsoft documents as the default for every process and thread,
  so it is a no-op unless something else lowered it. 1.4.1 enabled it and
  described it as a performance improvement; that was wrong. Now ships OFF in
  every profile, with the reason stated in code and INI.

- bDisablePowerThrottling (accepted). It genuinely opts the process out of
  EcoQoS execution-speed throttling, but that is a process-wide scheduling
  policy with no measured disk-cache benefit. Moved out of the default and
  into the Experimental profile only.

- RANDOM_ACCESS hint (partially accepted). Kept archive-only in Safe, since a
  BSA is read by seeking to internal blocks, but the Minimal profile now ships
  it OFF so users can run strip-NO_BUFFERING-only until it is benchmarked.

- Warm cache and DirectStorage remain OFF in every shipped profile.

Not accepted, with evidence:
- The reported hash "discrepancy" was two legitimate builds of the same commit.
  e39502a8/6fda7749 is the local VS2026 build; 5cb68bab/5ac34c5f is the CI
  VS2022 build attached to the GitHub release. The 1.4.1 release notes already
  stated MSVC output is not bit-identical across toolchains. To remove the
  ambiguity entirely, only the CI artifact is published from now on.

Also in this release:
- Versioning unified. Nexus 1.1.2/1.1.3 used a different scheme from the repo
  tags, making user reports impossible to map to a commit. DLL, git tag and
  Nexus file version are now the same number. Mapping recorded in
  MEMORY/NEXUS-VERSION-MAP.md: Nexus 1.1.2 = repo 1.3.0; Nexus 1.1.3 = a
  narrow IAT build whose source was never published (see below).
- One FOMOD now carries Safe / Minimal / Experimental instead of separate
  Nexus files, with the trade-offs written into the installer descriptions.
- All 1.4.1 correctness work retained: unfoldable policy self-test, pure
  (non-mutating) self-test, IsSafetyGated() shared by policy and counters,
  honest in_scope_safety_gated counter, stats reachable with the warmer off.

PROVENANCE WARNING recorded during this release:
  The Nexus 1.1.3 main file (SHA-256 8252ee1b...) is byte-identical to the
  192 KB DLL in NextGenDiskCache-1.4.0-Safety-Rework, which was flagged on
  2026-07-21 as not built from published source. It imports only KERNEL32 +
  dynamic UCRT (the repo source needs SETUPAPI + SHELL32 and links a static
  CRT), emits strings the repo source never contains, and no source for it
  exists in the repository or anywhere in the workspace. It cannot be rebuilt
  by anyone. 2.0.0 exists partly to replace it with an equivalent narrow build
  that IS reproducible from public source.

Verification (target rig, split-layout harness):
  - Safe (iHookScope=0): IAT hook installed into KERNEL32.dll entries of the
    main executable. EXE's own open .bsa 28000080 -> 10000080 (optimized);
    identical open issued from a separate DLL produced NO log entry.
  - Experimental (iHookScope=1): both opens intercepted (2 entries).
  - Self-test failure string present in the /O2 binary.
  - build.ps1 VS18 green; validate_rc.py PASS (now pins IAT-default,
    per-profile iHookScope, and the anti-folding tokens);
    package-release.ps1 gates PASS.
  - UNPROVEN: in-game SKSE session on 1.5.97 / 1.6.640 / 1.6.659 GOG, and any
    comparative benchmark of Safe vs Experimental.

NextGen Disk Cache 1.4.1
============================================
- Policy self-test is a real runtime guard again. The 1.4.0 version tested
  constants it assigned itself, so MSVC /O2 folded it to `true` and removed the
  failure branch entirely (verified: the error string was absent from the 1.4.0
  binary while control strings were present). It now checks safety invariants
  against the actually-loaded settings, uses volatile inputs so it cannot be
  constant-folded, and no longer mutates the live g_settings the hooks read.
- Re-enabled bDisablePowerThrottling and bRaiseMemoryPriority in the shipped
  SafeDefault and ExperimentalWarmCache profiles. Both are process-scoped
  (SetProcessInformation on GetCurrentProcess) and cannot affect other SKSE
  plugins, so the 1.4.0 hook-compatibility rationale did not apply to them.
  EcoQoS throttling measurably hurts a foreground game. Minimal profile keeps
  them off by design. bExpandWorkingSet remains off in every profile.
- Statistics counter renamed write_or_async_left_untouched ->
  in_scope_safety_gated and corrected: it counted every unmodified archive or
  asset open (including out-of-scope kinds) while claiming to count the
  write/async gate. It now counts only in-scope handles refused for safety.
- The stats snapshot is now emitted when the warm cache is disabled, via a
  one-shot low-priority worker. Previously it was produced only by the warm
  cache, so in the shipped default (warmer off) the counters were unreachable
  and could never appear in a user bug report.
- Extracted the safety gate into IsSafetyGated() so the flag policy and the
  statistics counter cannot drift apart.
- Validator pins the anti-folding fix (volatile self-test inputs, live-settings
  check, shared gate) so this cannot silently regress.

Verification (target rig, split-layout harness - exe in game root, plugin in
Data\SKSE\Plugins):
  - "internal file-policy self-test failed" now PRESENT in the /O2 binary
    (absent in 1.4.0) - the guard can actually fire.
  - Per-open flag transitions with bLogEveryOpen=1:
      .bsa  read + NO_BUFFERING + SEQUENTIAL  28000080 -> 10000080 (optimized)
      .bsa  read + NO_BUFFERING + OVERLAPPED  60000080 -> 60000080 (gated)
      .bsa  WRITE + NO_BUFFERING              20000080 -> 20000080 (gated)
      .esp  read + NO_BUFFERING + SEQUENTIAL  28000080 -> 28000080 (out of scope)
      .cosave read + NO_BUFFERING             20000080 -> 20000080 (save kind)
  - Process hints confirmed applied in log; exports/imports unchanged
    (SETUPAPI, KERNEL32, SHELL32; three SKSEPlugin_* exports only).
  - UNPROVEN: in-game SKSE session across 1.5.97 / 1.6.640 / 1.6.659 GOG.

NextGen Disk Cache 1.4.0 - Safety Rework
============================================
- Default hook scope is now BSA/BA2 archives only.
- Unknown extensions are never modified.
- Write-capable, create/truncate, overlapped, write-through, and delete-on-close opens are untouched.
- RANDOM_ACCESS is archive-only; loose assets, audio, plugins, INI/JSON, and databases retain caller hints.
- Added an internal policy self-test that disables hooks on failure.
- Warm cache and DirectStorage probing now ship disabled.
- Added a bounded experimental warmer: 512 MB, 8 MB/archive, one thread, 60-second delay.
- Warm cache scans only BSA/BA2, uses stable alphabetical coverage, and auto-tune cannot inflate configured limits.
- Numeric INI values are clamped to prevent unsigned wrap and unbounded work.
- Added Safe Default, Minimal Compatibility, and Experimental Bounded Warm Cache profiles.

## [1.3.1] - 2026-07-20
Agent: Claude
Intent: Fix DirectStorage runtime detection in the real game layout; restrict
        the CreateFile cache policy to buffered read handles.
Root cause (DirectStorage):
  - dstorage.dll carries no import entry for dstoragecore.dll; it loads it by
    bare name at first use. That lookup searches the application directory
    (the game root), System32, and PATH - never Data\SKSE\Plugins - so
    initialization failed hr=0x80004001 whenever the FOMOD-installed runtime
    sat beside the plugin. LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR cannot influence
    that dynamic load; it only scopes static import resolution.
Root cause (cache policy):
  - PatchFlags ignored dwDesiredAccess and completion/durability flags: write
    handles and OVERLAPPED / WRITE_THROUGH opens of asset-classed or unknown
    files were still patched. Same failure family as the 1.2.1 S.L.A.C.K.
    incident, but extension lists cannot anticipate every state file other
    software writes. Stripping NO_BUFFERING from an OVERLAPPED handle can also
    turn expected-async completions synchronous (latent use-after-free in
    fragile callers) - a plausible, unconfirmed vector for the reported
    end-of-dialogue crash. It also sabotaged DirectStorage's own unbuffered
    archive opens whenever raw reads were enabled.
Changes:
  - DirectStorageBackend: pre-load dstoragecore.dll by absolute path from the
    plugin directory before loading dstorage.dll; release it on shutdown;
    record core presence before factory creation; the failure log now names
    the missing core DLL explicitly.
  - PatchFlags receives dwDesiredAccess: any write/delete intent,
    FILE_FLAG_OVERLAPPED, or FILE_FLAG_WRITE_THROUGH leaves caller flags
    untouched regardless of extension. Read-side cache policy is unchanged.
  - New stats field write_or_async_left_untouched; README 1.3.1 section;
    version strings bumped across build/packaging/validator files.
Validation (2026-07-20, target rig; split-layout harness: exe in game root,
plugin + runtime in Data\SKSE\Plugins - the layout SKSE actually produces):
  - 1.3.0 repro first: "runtime found ... initialization failed hr=0x80004001".
  - 1.3.1 raw-read ON: "runtime ready ... queue created", 32 MB read via
    DirectStorage. Probe-only: "runtime validated ... queue not created".
    Runtime absent: PrefetchVirtualMemory fallback. All PASS.
  - Flag-gate self-test via bLogEveryOpen: write+WRITE_THROUGH+NO_BUFFERING
    .json A0000080->A0000080 (untouched); read-only .json 20000080->10000080
    (NO_BUFFERING stripped, RANDOM_ACCESS added); OVERLAPPED read
    60000080->60000080 (untouched).
  - build.ps1 VS18 Release green (only pre-existing vendored warnings);
    validate_rc.py PASS; package-release.ps1 gates PASS.
  - UNPROVEN: in-game SKSE runtime test. The crash report fix is a plausible
    vector removal, not a confirmed reproduction - crash log requested.

## [1.3.0] - 2026-07-18
Agent: Kimi
Intent: Manual game-drive class override for RAID / Storage Spaces / USB setups.
Root cause:
  - Drive class comes from seek-penalty + bus-type IOCTLs on the game volume.
    RAID0 NVMe, Storage Spaces, and some enclosures report a generic or RAID
    bus type, so fast arrays fall back to the SATA SSD profile (or the HDD
    profile if the controller reports a seek penalty): fewer warm readers and
    a smaller per-file cap than the hardware could sustain.
Changes:
  - New [Hardware] iGameDriveClass: 0 auto (default), 1 HDD, 2 SATA SSD,
    3 NVMe SSD. Applied right after detection and logged when it changes the
    detected class.
  - Override steers warm-cache auto-tune only (budget cap, per-file MB, reader
    threads). The CreateFile cache policy is drive-class independent.
  - Documented in all shipped INIs and README; validator pins the shipped
    default to auto.
Validation:
  - Full build + validate_rc.py + package-release.ps1 gates PASS.
  - Harness: iGameDriveClass=3 forces the NVMe profile; 0 leaves detection
    untouched (regression check).
